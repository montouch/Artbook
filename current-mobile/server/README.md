# Artbook Local API Scaffold

This is a plain Node.js development scaffold for turning the Artbook prototype into a server-owned app. It uses JSON storage in `server/data/dev-store.json` so it can run without npm install, Docker or cloud credentials.

Auth is still a development scaffold, but demo passwords are hashed with PBKDF2-SHA256, login requires a password, and bearer sessions include expiry timestamps. Do not treat this as a substitute for production auth, MFA, device risk or account recovery.

Run:

```powershell
node server/src/server.mjs --check
node server/src/server.mjs
```

Default URL: `http://127.0.0.1:8787`

Public privacy web scaffold:

- `GET /privacy-policy` serves a host-ready privacy policy draft covering account/profile, marketplace, bookings, messages, location, provider-led payments, identity/KYB, AI assistance, retention and deletion.
- `GET /account-deletion` serves a host-ready account deletion request page for users who cannot use the Android app.
- `POST /api/public/deletion-requests` records a review-only public deletion request without requiring app auth.
- These routes do not delete production data and are not final legal copy. Privacy Support must verify ownership, retention blockers and provider deletion before action. Host these pages on the production Artbook domain, complete legal/provider review, and enter the privacy/deletion URLs in Play Console before launch.

Provider-backed flows intentionally fail closed until credentials and webhooks exist:

- KYC/identity
- payments and mobile money
- media/object storage
- calls
- external delivery webhooks
- music distribution, copyright/CMO filing and release identifiers

Supabase launch backend pack:

- `../supabase/migrations/20260605014500_artbook_launch_core.sql` is a concrete Supabase/Postgres migration for the launch core: accounts, memberships, profiles, posts, listings, bookings, booking events, orders, messages, provider event replay, trust evidence, support cases, AI tasks, outbox and audit events.
- Every public table in that migration has RLS enabled and forced. Policies are account/tenant scoped through `auth.uid()` and `artbook_account_memberships`, with private helper functions kept in `artbook_private`.
- `../supabase/functions/provider-webhook/index.ts` is a fail-closed Edge Function for provider callback replay. It reads the raw body, verifies `ARTBOOK_PROVIDER_WEBHOOK_SECRET`, stores only a payload digest/safe shape metadata through the service role, and never stores raw provider payloads or moves money.
- `../tools/supabase-launch-backend-audit.mjs` verifies the migration/function boundaries: 15 tables, RLS on all tables, 42 policies, no client provider-event writes, no raw payload storage, no money movement and no Android creator monetization.
- The connected Supabase account has an organization but no project visible during this pass, so these artifacts are GitHub-ready and not live-applied.

`POST /api/calls` is a masked-relay contract, not a direct dialer. It now requires an active server-owned work context, proves the caller and peer are parties to that record, blocks missing/foreign/inactive contexts before provider handoff, includes expiry and rate-limit policy, and still returns `provider_not_configured` without echoing caller/callee phone numbers or calling a telephony provider.

AI endpoints are backend-owned. `POST /api/ai/context-preview` and `POST /api/ai/business-brief` build redacted visible-record previews and keep protected actions blocked. `POST /api/ai/live-assist` can call OpenAI only when `OPENAI_API_KEY` is present in `.env.local`/the server environment and `ARTBOOK_AI_LIVE=1`; the Android APK never receives the key. Provider errors or quota/billing failures fail closed to a local guarded brief, and money movement, identity approval, Provenance Seals, restricted media publishing, moderation decisions and settlement transitions remain blocked.

Play Billing is backend-owned review evidence, not a local entitlement grant. `GET /api/play-billing/entitlements` gives Review Ops a redacted product catalog, Android-vs-web-only split, required secret names, purchase-token review rows and RTDN replay rows. Authenticated Android clients can post a Play purchase result to `POST /api/play-billing/purchase-token-reviews`, but the scaffold stores only a SHA-256 purchase-token digest, never the raw token, and returns `purchase_token_review_only_no_entitlement_grant` until Google Play Developer API verification, acknowledgement, restore/cancel/refund rules and RTDN handling are real. `POST /api/play-billing/rtdn-events` records replay-only RTDN summaries for Review Ops and never changes entitlement, wallet, payout, refund, escrow or founder revenue state. Restricted creator monetization remains web-only outside the Play Store Android app.

Provider-led payment boundaries are also backend-owned review evidence. `GET /api/payments/provider-boundary` exposes the fail-closed rail map for M-Pesa, cards, jobs/bookings escrow, internal transfer ledger, courier/seller payouts and founder fee reporting. Review Ops can record non-settling boundary rows at `POST /api/payments/provider-boundary-events`; the server stores provider references as SHA-256 digests only, never calls providers, never credits spendable balances, never releases escrow/payouts, never grants digital entitlements and never recognizes founder revenue. `GET /api/founder/finance-export` returns a copy-ready founder finance export for Play subscriptions, marketplace service fees, booking tools, freelancer escrow, delivery commissions, wallet transfer fees and boosts; it reports gross reference volume, estimated founder revenue and recognized founder revenue as separate fields and keeps recognition at zero until Play Billing payout reports or provider reconciliation prove it. The export now also includes a double-entry preview that debits pending provider/Play receivables and credits unearned founder-fee clearing, never revenue accounts, plus refund-window, chargeback/dispute and tax/accounting holds before recognition. It also includes a refund/chargeback hold export that lists provider-unverified refund, support, dispute, receipt-candidate and webhook replay signals, then blocks founder revenue until provider return proof, support outcome, reversal/credit-note journal and tax review are complete. The same export includes a production ledger and partner handoff for payment reconciliation, double-entry ledger, escrow/wallet/payouts, Play Billing revenue, tax/accounting and support/dispute workstreams; all implementation fields are server/provider-owned with zero client-writable ledger or provider fields. That handoff now maps the fields into non-settling endpoint contracts and production database tables for provider webhooks, provider fetch proof, receipt reconciliation, ledger previews/posts, escrow/payout review, wallet provider settlement, Play token verification, tax export batches and support dispute decisions. It also includes a review-only Postgres/Supabase-style migration SQL blueprint with `artbook_money_ops` tables, server-only roles, fail-closed RLS policies, client-write checks and money-movement checks; the app never applies that SQL. Worker job contracts map those endpoints and tables into disabled server-only queues with idempotency keys, retry/dead-letter policies and zero client-runnable or money-moving workers. Route schema contracts now define the server-role headers, idempotency requirements, request fields, accepted response fields and rejection codes for the worker-backed routes while still blocking route runners, client writes and money movement. `GET /api/settlements/state-machine` exposes the Review Ops settlement reconciliation path from client replay, provider webhook replay, webhook review decisions, provider fetch proof, receipt candidates and reconciliation previews to the still-blocked production settlement terminal states. Physical goods, bookings, jobs, deliveries and wallet transfers stay on provider-led rails; Android digital subscriptions stay in the Play Billing entitlement lane.

Wallet replay endpoints store local prototype ledger rows as `client_replayed_not_settled` audit data. They accept only rows where the authenticated profile is explicitly present in `from`, `to`, `account` or `parties`. Escrow settlement audit endpoints do the same for Fundi/order release and refund claims as `client_replayed_audit_only_not_settled` rows, requiring an involved party, record id and positive amount. The settlement exception queue keeps provider-unverified releases, refunds and support holds visible with work-evidence summaries, provider-receipt placeholders, support status and a non-settling support timeline until real reconciliation clears them; Review Ops/support can append review notes at `POST /api/settlements/exceptions/:id/review-notes`, record receipt candidates at `POST /api/settlements/exceptions/:id/receipt-candidates`, and preview amount/currency/party mismatches at `GET /api/settlements/exceptions/:id/reconciliation-preview`, but those records are non-settling audit/preview data and preserve provider-unverified, non-spendable status. `GET /api/providers/readiness` lets Review Ops/support inspect provider readiness by secret name/status only, raw-body signature readiness, replay-store scaffold readiness, production runtime/deployment readiness, Play Store blockers and a backend/Android/compliance/payments release checklist; it never returns secret values or enables settlement. The runtime deployment section groups backend, payments, delivery/call relay, Play/Android and compliance environment names with sandbox callback checks, a six-step deployment runbook, an eight-row backend hosting checklist, local sandbox fixture execution evidence contracts, fixture result capture rows that point to the replay ledger and review-decision endpoints, and a backend deployment evidence packet for host, secret-store, raw-body, observability, backup, rollback, Android API and provider contract proof. Local replay evidence is explicitly marked as not production-host proof, and all deployment, provider activation, provider calls, fixture runners, receipt creation, dispatch, entitlement grants and money movement flags remain false. `GET /api/providers/readiness/export` returns the same boundary as a redacted copy-ready plain-text snapshot for backend or Play Store checklists. `GET /api/providers/readiness/evidence-packet` bundles that snapshot with the owner checklist summary, sandbox fixture result capture summary, backend deployment evidence summary, current local APK hash/build metadata and latest logged audit results for Review Ops handoff, still as `release_evidence_packet_review_only_no_settlement`. Backend sync checklist ticks are local Review Ops notes only and do not configure providers, clear blockers or enable money movement. `GET /api/settlements/webhooks/:provider/fixture-templates` returns M-Pesa/Daraja, card checkout and payout-rail fixture payloads with raw-body signature and replay-protection notes as templates only. `GET /api/settlements/provider-fetch/:provider/proof-stub` returns the server-side secret checklist, request contract, response proof fields, replay keys and blocked settlement transitions for M-Pesa/Daraja, card checkout and payout fetches, without calling providers or storing proof. `POST /api/settlements/webhooks/:provider` can map common provider/M-Pesa/card/payout payload fields into a dry-run candidate and preview, record a separate replay-ledger event summary with payload digest/shape, signature state and idempotency decision, then still returns `provider_not_configured`; it never stores the callback as a receipt candidate or changes payout/refund/spendable state. `GET /api/settlements/webhook-events` exposes those replay-only events to Review Ops/support, and `POST /api/settlements/webhook-events/:id/review-decisions` classifies them as duplicate, signature-invalid, needs-provider-fetch or ready-for-receipt-candidate without creating candidates, provider success, payout/refund state or spendable funds. Duplicate provider/idempotency keys are recorded as duplicate replay events, not as settlement events. These endpoints are useful for backend handoff and QA, but they are not a real wallet, provider settlement, KYC decision, payout approval or source of spendable funds.

The Android Backend Sync desk also keeps a production deployment proof intake. Review Ops can attach redacted host/provider artifact notes to the same backend deployment evidence lanes; when the backend is connected those notes are stored through `POST /api/providers/deployment-evidence-notes` and listed through `GET /api/providers/deployment-evidence-notes` as server-held review evidence. The endpoint stores a sanitized source label plus SHA-256 source digest, never raw credential material, private numbers, raw identity documents or raw restricted media. These notes are still handoff evidence only: they do not approve deployment, configure providers, call payment rails, dispatch couriers, grant entitlements, release escrow, credit wallet balances, complete refunds or recognize founder revenue.

Country Passport endpoints store the current profile's legal-ID country, operating country, payout/tax country, real device GPS country proof, residence proof, work permission and source-of-funds intent for review. `POST /api/identity/ai-verification-drafts` can create a redacted draft and missing-evidence list from that packet, but it always returns `canApprove:false`, `providerRequired:true` and blocked protected actions. Artist music release packet endpoints store rights, credits, split, sample/beat, artwork, master-quality, metadata, identifier/admin and artist-approval evidence; even an artist-approved complete packet is only `ready_for_provider_review` and cannot file copyright/CMO paperwork, distribute music or enable royalties without real providers and human review.

Order and booking endpoints create server-owned work records for provenance testing. Seller/provider proof is not enough by itself: a Provenance Seal or active trust report can reference `order:<id>` or `booking:<id>` only after buyer/booker-confirmed proof is recorded and both parties match the authenticated user and target profile.

Trust reports are guarded against score manipulation: repeated reports from the same reporter/evidence are rejected, and a report that contradicts an existing Provenance Seal is held in non-scoring moderation review until a real admin workflow resolves it.

The scaffold includes a first moderator-only trust queue at `GET /api/moderation/trust-reports` and resolver at `PATCH /api/moderation/trust-reports/:id`. After a moderator requests more evidence, involved parties can add a note and optional verified order/booking proof through `POST /api/trust/reports/:id/evidence-responses`; the report stays non-scoring until a moderator accepts, rejects or asks again through `PATCH /api/moderation/trust-reports/:id/evidence-responses/:responseId`. If accepted proof later leads to dismiss/uphold/escalate, the final report resolution is labeled with the accepted response id so proof acceptance and scoring/no-scoring resolution remain separate audit events. Local `admin`/`moderator` profile roles or `ARTBOOK_MODERATOR_PROFILE_IDS` can access the queue; real production should replace that with provider-backed staff auth, audit review and separation of duties.

Restricted-media safety reports are server-owned metadata only. `POST /api/restricted-media/reports` records viewer/creator/vault/watermark notes, opens an urgent support incident and rejects raw media uploads in this scaffold. Review Ops can inspect `GET /api/moderation/restricted-media-reports` and stage `PATCH /api/moderation/restricted-media-reports/:id` decisions such as more evidence, temporary hold, dismissal or escalation, but media provider action remains fail-closed until real storage, takedown, age/ID, consent and legal/safety operations exist.

Set `ARTBOOK_FAKE_PROVIDERS=1` only for local backend demos where fake provider success is desired.
